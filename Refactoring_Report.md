# Refactoring Report: Warehouse System (Draft)

## 1. Нарушения принципов SOLID и архитектурные проблемы текущего кода

### **Violation of Single Responsibility Principle (SRP)**
В текущей версии файл `views.py` (и конкретно функция `warehouse_view`) берет на себя слишком много ответственностей:
- **Чтение и Запись файлов (I/O)**: Прямая работа с файловой системой (`read_json`, `write_json`, `open`).
- **Авторизация**: Проверка логинов и паролей.
- **Бизнес-логика CRUD**: Добавление, редактирование, удаление из сырых списков `warehouse_data.append()`.
- **Финансовая математика**: Прямо во view считаются налоги `tax = 0.20` и скидки.
- **Отчетность**: Генерация строчек для TXT-отчета и запись файла отчета происходит перед рендерингом HTML.

### **High Coupling (Сильная связность)**
- Бизнес-логика жестко привязана к структуре JSON-файла. Малейшее изменение формата данных потребует переписывания логики контроллера.
- Функции покупок жестко связаны с путями к лог-файлам и файлам отчетов.

### **Характерный "Hardcode"**
- Правила маржинальности, налогообложения и скидок («если электроника — налог 20%») прошиты непосредственно в теле функции `warehouse_view` через массивные `if-elif`.

## 2. План рефакторинга с использованием паттернов проектирования

В следующей версии мы исправим данные проблемы с помощью следующих архитектурных паттернов:

### **Pattern 1: Strategy (Стратегия)**
**Цель:** Избавиться от массивного блока `if-elif` при расчете налогов и скидок.
- **Как это будет реализовано**: Будет создан интерфейс (базовый класс) `PricingStrategy` с методом `calculate_final_price(price)`.
- Для каждой категории (Electronics, Food, Clothing и т.д.) будут написаны конкретные стратегии: `ElectronicsPricingStrategy`, `FoodPricingStrategy`.
- Контроллер будет просто вызывать `strategy.calculate_final_price()`, не зная, какие там проценты налогов и скидок зашиты внутри конкретного класса. Легко расширять в будущем (Open/Closed Principle).

### **Pattern 2: Facade (Фасад)**
**Цель:** Скрыть сложную логику операции "Закупки" (Обновление остатков + логирование + пересчет стоимости склада + запись в отчет) от контроллера.
- **Как это будет реализовано**: Мы создадим сервисный класс `PurchaseFacade`, у которого будет единственный метод `execute_purchase(item_id, user, qty)`.
- Контроллер (view) будет только вызывать `PurchaseFacade().execute_purchase(...)`. 
- Внутри фасада будут вызываться подсистемы (WarehouseRepository для обновления остатков, LoggerService для записи логов, ReportService для изменения отчетности). Это кардинально очистит `views.py`.

### **Pattern 3: Repository (Бонус для работы с JSON)**
**Цель:** Абстрагировать CRUD-операции над JSON.
- **Как это будет реализовано**: Создание класса `JsonWarehouseRepository`, который будет иметь методы `get_all()`, `add(item)`, `update(item)`, `delete(item_id)`. Контроллеры не должны знать, что используется JSON — они будут запрашивать данные только через интерфейс репозитория.

### **Итог**:
Эти изменения сделают код поддерживаемым, расширяемым и соответствующим SOLID-принципам, а главное — полностью разделят слой HTTP-взаимодействия (views) и слой бизнес-логики.
